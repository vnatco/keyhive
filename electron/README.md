# KeyHive Desktop App

Desktop wrapper for [KeyHive](https://keyhive.app) using Electron. Loads the web frontend locally inside a frameless window with native OS integrations.

## Files

| File | Purpose |
|------|---------|
| `main.js` | Main process -window management, system tray, global shortcut, security policies, custom `app://` protocol |
| `preload.js` | Bridge between main and renderer -exposes IPC for window controls and settings |
| `run-windows.cmd` | Windows quick-launch -auto-installs dependencies and starts the app |
| `build-windows.cmd` | Windows shortcut to build the installer (`npm run build:win`) |
| `icons/` | App and tray icons (SVG, PNG at multiple sizes, ICO) |
| `www/` | Web frontend -generated by the root `build.js`, do not edit directly |
| `installer/` | Custom NSIS scripts for the Windows installer |

## How It Works

The app registers a custom `app://local` protocol that serves files from `www/`. This is required because `file://` has a `null` origin which blocks Web Workers and crypto operations.

The web app calls `enableDesktopMode()` (defined in `public/js/utils/platform.js`) which activates desktop-specific behavior:

- **Custom title bar** -replaces the native frame with refresh, minimize, and close buttons
- **System tray** -app icon lives in the tray. Right-click for context menu.
- **Global shortcut** -`Ctrl+Alt+Z` (customizable in Settings > Desktop App) toggles the window from anywhere
- **Run on startup** -uses the OS native auto-launch mechanism, starts hidden in tray
- **Single instance lock** -a second launch focuses the existing window instead of opening a new one
- **External links** -`http://https://` URLs open in the default browser; all other schemes are blocked
- **Window state** -position and size are saved to `settings.json` in the OS user data directory

### Security

- Renderer is sandboxed (`sandbox: true`, `contextIsolation: true`, `nodeIntegration: false`)
- Content Security Policy set via response headers in `main.js` (single source of truth)
- Keyboard shortcuts blocked: refresh, DevTools, view source, zoom, print, find, address bar
- Path traversal protection on the `app://` protocol handler
- External navigation blocked inside the app window

## Getting Started

### 1. Build the frontend

From the root of the repository, run:

```bash
node build.js electron
```

This copies the contents of `web/` into the `electron/www/` folder and patches paths for local loading.

### 2. Install dependencies

```bash
npm install
```

### 3. Run the app

```bash
npm start
```

On Windows, you can simply double-click **`run-windows.cmd`** -it will install dependencies automatically if needed and start the app.

## Building Installers

Build distributable packages for each platform:

```bash
npm run build:win    # Windows (.exe installer) -output in dist/
npm run build:mac    # macOS (.dmg)
npm run build:linux  # Linux (.AppImage)
```

On Windows, you can also double-click **`build-windows.cmd`** to build the installer.

> To skip code signing (if you don't have a certificate), set `CSC_IDENTITY_AUTO_DISCOVERY=false` before building.

## Installer Behavior

- Installs to `C:\Program Files\KeyHive` (per-machine, requires admin)
- Creates desktop shortcut and Start Menu entry
- User can choose install directory
- **Uninstaller** asks whether to keep or remove app data (`%AppData%/keyhive/`)
- App data (settings, cached vault) is preserved across updates by default

## User Data

Stored in `%AppData%/keyhive/` (separate from install directory):

- `settings.json` -window state, global shortcut, run-on-startup preference
- `IndexedDB/` -encrypted vault cache (critical for local-mode users)

This data survives app updates and reinstalls. Only removed if the user explicitly chooses to during uninstall.
